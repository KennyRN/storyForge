# storyForge code audit

Full-codebase pass over `src/` (~6,050 lines, 35 files) plus `manifest.json`/`package.json`/build config, 2026-07-09. Every finding below was traced to a specific file/line and, for the highest-impact claims, independently re-verified by grepping/reading the actual source (not just taken from the initial pass). Trivial style/formatting nitpicks are excluded.

Each entry cites `file:line`, what's wrong, why it matters, and — where relevant — a fix or the tradeoff options.

## 2. Confusing or misleading code

- **Two settings state stores that don't agree** — [main.ts:64](src/main.ts#L64) persists `collapsedSections` but [StoryForgeSettingsTab.ts:10,74](src/view/StoryForgeSettingsTab.ts#L10) maintains its own in-memory `expandedSections` Set that's reset to empty every time `display()` runs (confirmed at line 74: `this.expandedSections = new Set();`). **Practical effect: every foldable settings section collapses shut each time the user reopens Settings**, even if they left it expanded last time. Fix: read/write `collapsedSections` from plugin settings instead of a fresh in-memory Set.
- **A settings section that renders but does nothing** — [StoryForgeSettingsTab.ts:167-173](src/view/StoryForgeSettingsTab.ts#L167-L173): "Text Style" and its five nested subsections ("Editor Text", "Heading 1/2/3", "Other Headers") are wired up with empty no-op render callbacks. A user expanding any of them sees a header and a chevron flip with nothing inside — reads as a bug rather than an intentional placeholder, and there's no comment flagging it as WIP.
- **Icon identifiers that don't match their SVG** — [icons.ts:3](src/icons.ts#L3): `ICON_UNPLACED = "sf-archive-drawer"` is registered with `INBOX_SVG` (an inbox glyph, not a drawer). [icons.ts:5](src/icons.ts#L5): `ICON_UNARCHIVE = "sf-inbox-download"` is registered with `ARROW_OUT_UP_SQUARE_SVG` (an arrow, not an inbox). A future maintainer renaming/swapping one based on the name alone would likely pick the wrong glyph.
- **`formatSingleLine` can produce a bare leading colon** — [titleNumbering.ts:42-45](src/titleNumbering.ts#L42-L45): its doc says it produces `"Title: Subtitle"`, but if `splitTitleSubtitle` yields an empty title (e.g. raw input `"// Subtitle only"`), the output is `": Subtitle only"`. See bug entry below.
- **Undocumented reliance on a private Obsidian field with no fallback signal** — [tabTitles.ts:12-14](src/tabTitles.ts#L12-L14): `LeafWithTabHeader.tabHeaderInnerTitleEl` is an unofficial internal; the call site silently no-ops if it's ever renamed. Documented as unofficial in a comment, but there's no warning if it stops working — a future Obsidian update could silently disable tab-title overrides with zero signal to anyone.
- **Double re-render path on archive** — [TopPanel.ts:236-240,270-274](src/view/TopPanel.ts#L236-L240): archiving a chapter directly calls `renderTopPanel(...)` *and* invokes `options.onArchiveChapter?.()`, which in [StoryForgeView.ts:108-113](src/view/StoryForgeView.ts#L108-L113) can itself trigger a full `render()` via `refreshStats()`. Harmless but confusing to trace, and wasteful.

## 3. Potential bugs

- **Shared word-count file has an unguarded read-modify-write race** — [history.ts:30-41](src/history.ts#L30-L41) (`upsertTodayTotal`) reads the single shared `wordcount.md` (all books' totals in one file), updates one book's total, and writes the whole file back. It's invoked from two independent, unsynchronized triggers: [main.ts:424-438](src/main.ts#L424-L438) (debounced per-chapter on vault `modify`) and [StoryForgeView.ts:196-203](src/view/StoryForgeView.ts#L196-L203) (on user actions like archive/unarchive). If edits land close together for *two different books*, both read the whole file, compute their own book's total against a stale snapshot, and the second write clobbers the first book's just-written total. Fix: serialize writes behind a queue, or use `Vault.process()` for an atomic read-modify-write.
- **Inconsistent fallback frontmatter in `renameChapterEntry`** — [book.ts:349-371](src/book.ts#L349-L371), verified directly: every other book.md writer (`writeBookChapterOrder`, `archiveChapter`, `unarchiveChapter`, `upsertChapterEntry`, `renameChapterTitle`) passes `defaultBookContent(bookId, bookTitle, position)` ([book.ts:46-50](src/book.ts#L46-L50)) as the fallback if book.md doesn't exist yet — seeding `book-id-reference`/`book-title-reference`/`series-order-reference`/`chapter-order`. `renameChapterEntry` instead passes the bare legacy literal `` `---\norder:\n---\n` `` with none of those fields and the old `order` key instead of `chapter-order`. Confirmed this literal is exactly what's in the code today. If this fires before book.md exists (e.g. a chapter is renamed via the OS file explorer before the plugin has created book.md for that folder), it creates a malformed book.md missing the reference fields other code paths expect. Fix: reuse `defaultBookContent(...)` here too.
- **Same stale-literal pattern in migration.ts, currently dormant** — [migration.ts:133,152](src/migration.ts#L133) also pass the bare legacy literal, but both call sites are guarded by an early-return if the file doesn't already exist, so it's provably dead there today. Worth aligning anyway — a future edit that loosens that guard would silently reintroduce the bug from the point above.
- **`formatSingleLine` empty-title edge case** — [titleNumbering.ts:29-45](src/titleNumbering.ts#L29-L45): `splitTitleSubtitle("// A Subtitle")` returns `{ title: "", subtitle: "A Subtitle" }`; `formatSingleLine` only special-cases an empty *subtitle*, not an empty *title*, so it returns `": A Subtitle"`. Not covered by `titleNumbering.test.ts` (all subtitle tests use a non-empty title). Fix: `return subtitle ? (title ? \`${title}: ${subtitle}\` : subtitle) : title;`
- **Non-Latin titles collapse to the same generated ID** — `slugify` in [slug.ts](src/slug.ts) strips everything outside `[a-z0-9]` after lowercasing, so any non-Latin-script title (Cyrillic, CJK, Japanese, Arabic, emoji-only) collapses to the literal fallback `"book"`. `mintId` de-duplicates via `-2`, `-3`… suffixes so it won't overwrite, but every non-Latin book/series in a vault ends up named `book`, `book-2`, `book-3`… with no trace of the original title. `bookCode.ts`'s `titleToId`/`segment()` has the same ASCII-only assumption and produces literal `"XXX"` for an all-non-ASCII title. Untested (existing test only covers ASCII punctuation-only input). Real correctness/UX gap for any non-English author.
- **Drag-and-drop writes fail silently** — [TopPanel.ts:211-213,296-298](src/view/TopPanel.ts#L211-L213), [BottomPanel.ts:93-97](src/view/BottomPanel.ts#L93-L97), [SeriesModal.ts:64-66](src/view/SeriesModal.ts#L64-L66): all four reorder-persist calls (`reorderSeriesBooks`, `writeBookChapterOrder`, `moveCodexItem`) are fired with `void ...(...)`, and the DOM has already been optimistically reordered. If the vault write throws, the rejection is swallowed with no `Notice` — unlike the sibling `handleCreateChapter`/`handleCreateBook`/etc. in the same files, which do wrap writes in try/catch + `Notice`. Result: the on-screen order can silently diverge from what's actually persisted. Fix: wrap in try/catch and surface a `Notice`, matching the create-handlers' pattern.
- **No `onClose()` on the main view — pending async work can run after the view is torn down** — verified directly: [StoryForgeView.ts](src/view/StoryForgeView.ts) defines `onOpen()` (line 51) but no `onClose()`. `this.debouncedRender` is backed by a `setTimeout` ([debounce.ts](src/debounce.ts)) that's never cancelled; closing the panel while a debounce is in flight (e.g. right after a vault `modify` event) still fires `render()` ~400ms later against a torn-down view. `refreshStats()` ([StoryForgeView.ts:155-193](src/view/StoryForgeView.ts#L155-L193)) does multiple `await vault.read(...)` calls before calling `render()` — if the view closes mid-await, it still runs `render()` on a closed view afterward. Contrast with `ToolsView`, which correctly defines `onClose()` ([ToolsPanel.ts:34-37](src/view/ToolsPanel.ts#L34-L37)) to reverse its side effects. Fix: add `onClose()` that sets a "closed" flag checked before `render()`/`refreshStats()` proceed, or cancel the pending timer.
- **Mid-drag re-render can orphan the dragged DOM node** — [dragReorder.ts](src/view/dragReorder.ts)'s drag handlers hold pointer capture across the whole gesture, but `StoryForgeView.render()` fires on a 400ms debounce in response to *any* vault event (including ones unrelated to the panel being dragged in), and calls `container.empty()`. If that fires mid-drag, the dragged row is detached from the document while pointer capture is still active; the drag continues against a stale/orphaned subtree, and `onUp` reads back stale rows. No mechanism suspends re-render during an active drag. Same risk applies to `dragReorderTree.ts`'s `elementFromPoint` hit-testing.
- **Negative/unclamped word-count deltas** — [historyMath.ts:10-18,41-50](src/historyMath.ts#L10-L18) (`wordsToday`/`wordsThisWeek`) compute `today - mostRecentPriorTotal` with no floor. If a user deletes more text than they add, this goes negative and [StoryForgeView.ts:178-179](src/view/StoryForgeView.ts#L178-L179) feeds it straight into the displayed stat — likely showing "-340 words today". May be intentional (accurate delta) but if the UI is meant to show only forward progress, needs `Math.max(0, …)`.
- **Duplicate ordering keys are silently dropped with no signal** — [ordering.ts](src/ordering.ts)'s `resolveOrder` builds a `Map` keyed by `keyOf(member)`; if two members ever produce the same key, the earlier one is silently discarded with no orphan/error signal. Low-probability on a real filesystem (filenames are unique) but undocumented and untested.
- **`recomputeDebouncers` map grows unbounded for the life of a session** — [main.ts:121,413-422](src/main.ts#L121): entries are keyed per chapter path and never pruned when a chapter is deleted/renamed. Harmless since the whole plugin instance is discarded on unload, but grows for as long as the plugin runs. Minor.
- **`initializeVaultState()` isn't awaited before views can render** — [main.ts:174-181](src/main.ts#L174-L181): `void this.initializeVaultState()` runs migration/reconciliation without being awaited, so an already-open view could paint its first frame before series.md/book.md exist or schema migration has run. Self-heals within ~400ms via the debounced re-render once `metadataCache` catches up, so impact is a brief flash of a partial panel rather than a lasting bug — but worth a comment noting this is an intentional startup-latency tradeoff.

## 4. Convoluted code — simplifiable now (with sketch)

- **5x duplicated "resolve book defaults" boilerplate in book.ts** — [book.ts:158-161,176-179,200-203,301-304,326-329](src/book.ts#L158-L161) all repeat:
  ```ts
  const entry = getSeriesBookEntry(app, bookFolderName);
  const bookId = entry?.bookId ?? mintId(bookFolderName, collectAllBookIds(app));
  const bookTitle = entry?.bookTitle ?? bookFolderName;
  const position = getSeriesOrderPosition(app, bookFolderName);
  ```
  This is exactly the pattern that was skipped in `renameChapterEntry` (see bug above) — the one place it wasn't duplicated is where the inconsistency crept in. Extract `resolveBookDefaults(app, bookFolderName)` and call it from all 6 sites (including `renameChapterEntry`), so `defaultBookContent(...)` is always constructed the same way.
- **Duplicated default-content constants across files** — `DEFAULT_SERIES_CONTENT` is independently defined with an identical literal in both [series.ts:19](src/series.ts#L19) and [migration.ts:8](src/migration.ts#L8); `DEFAULT_CODEX_CONTENT` likewise in both [codex.ts:43](src/codex.ts#L43) and [migration.ts:9](src/migration.ts#L9). If the default schema shape ever changes, it's easy to update one copy and forget the other — reintroducing the same class of bug as the `renameChapterEntry` finding. Fix: export from `series.ts`/`codex.ts` and import into `migration.ts`.
- **Duplicated word-count recompute logic** — [StoryForgeView.ts:196-203](src/view/StoryForgeView.ts#L196-L203) (`recomputeWordCount`) duplicates the tail of `recomputeChapter` in [main.ts:424-438](src/main.ts#L424-L438): both read all live chapter files for a book, sum word counts, and call `upsertTodayTotal`. Only `main.ts`'s version also maintains the fingerprint sidecar. Factor "recompute + persist today's total for a book" into one function both call — this also shrinks the surface area for the word-count race above.
- **4x duplicated ordered/unplaced row-rendering in TopPanel.ts** — `renderSeriesList` (lines 164-177, 185-198) and `renderBookList` (lines 223-249, 257-283) each render near-identical row markup/click/rename wiring twice, differing only in source array/container/index offset. A single `renderTitledRow(container, key, numberedTitle, onClick, renameOptions)` helper called 4x would collapse ~60 duplicated lines into ~12.
- **Duplicated mutually-exclusive-toggle pattern in the settings tab** — [StoryForgeSettingsTab.ts:320-347](src/view/StoryForgeSettingsTab.ts#L320-L347) (Unplaced Items: "Muted" vs "Header Colour") and [:505-532](src/view/StoryForgeSettingsTab.ts#L505-L532) (Codex Note Label: "Use Default Colour" vs "Use Folder Colour") independently implement identical "turning one toggle on forces the other off" logic. A small `bindExclusivePair(a, b, onChange)` helper would remove ~25 duplicated lines and prevent the two call sites drifting apart if the exclusivity logic ever needs a tweak.
- **Header-style setting group repeated near-verbatim** — [StoryForgeSettingsTab.ts:237-286](src/view/StoryForgeSettingsTab.ts#L237-L286) (Unplaced: size/colour/muted/small-caps) and [:376-424](src/view/StoryForgeSettingsTab.ts#L376-L424) (Codex: same four settings) could become one `renderHeaderStyleGroup(body, config)` parameterized by setting keys — see the scaling concern below for why this is worth doing before a third panel type is added.

## 5. Convoluted code that resists simplification (with rationale)

- **`resolveCodexTree`** ([codexTree.ts:77-108](src/codexTree.ts#L77-L108)) reconciles three overlapping notions — virtual folder structure, which files still physically exist, and which files are visible in the current book scope — while also maintaining parity with a since-removed real-folder-based UI. The branching is inherently complex because it models a virtual/logical tree layered over a flat real-file directory with independent "existence" and "visibility" axes. The existing comment already explains *why*, not just *what* — leave it, but any future change here needs a matching update to `src/__tests__/codexTree.test.ts` since many edge cases are encoded in prose rather than types.
- **Hand-rolled frontmatter parsing in `chapterSidecar.ts` instead of `metadataCache`** ([chapterSidecar.ts:10-18,45](src/chapterSidecar.ts#L10-L18)) — deliberate tradeoff: `updateChapterFingerprint` needs the *just-read* raw content's frontmatter, not `metadataCache`'s, which lags a write by an unspecified amount. Options: accept as-is (reasonably safe, narrow regex over a known-shape file), or add a one-line comment explaining why `metadataCache` wasn't used here, unlike the sibling `readChapterSidecarFrontmatter` right above it which does use it.
- **Two different drag-and-drop strategies (`dragReorder.ts` vs `dragReorderTree.ts`)** — justified by the file's own comment: flat lists can move real DOM nodes live during drag, but the Codex tree can have collapsed folders whose children aren't even in the DOM, so it needs `elementFromPoint` hit-testing plus persist-and-re-render on drop instead. Unifying these would either force the flat-list case to pay the tree's re-render cost, or force the tree case into a live-DOM-move model it can't structurally support. Keeping them separate is the right call — the tradeoff is that the mid-drag re-render fragility (§3) has to be fixed twice.
- **`StoryForgeSettingsTab.display()` as one ~590-line method** ([:70-661](src/view/StoryForgeSettingsTab.ts#L70-L661)) is large because it's a flat sequence of independent one-off UI blocks, each closing over `this.plugin`/`settings`/sibling toggle instances for cross-field logic. Splitting into per-section private methods (`renderPaletteSection`, `renderUnplacedSection`, etc.) is a mechanical extraction, not a conceptual simplification — the value is organizational (see scaling concern below), not logic reduction.
- **Custom collapsible-section headers instead of `Setting().setHeading()`** ([StoryForgeSettingsTab.ts:17-47](src/view/StoryForgeSettingsTab.ts#L17-L47)) — `setHeading()` only styles a row as a heading; it has no collapse/expand behavior. Genuine collapsible sections aren't possible via the public `Setting` API today, so this custom scaffolding is justified. The real fix available is persisting `expandedSections`/`collapsedSections` properly (§2), not removing the custom mechanism.

## 6. Future-proofing concerns

- **English/Latin-script assumption baked into identity generation** — `slug.ts`'s `slugify` and `bookCode.ts`'s `titleToId` both assume ASCII/Latin titles (see bug above). If "novel-writing workspace" ever explicitly means to support non-English authors, this is the first thing that will surface as a bug report. Options: (1) use Unicode-aware matching (`\p{L}\p{N}` property escapes) instead of ASCII-only ranges; (2) at minimum, fall back to a counter-based id (`untitled-1`, `untitled-2`) rather than the same literal `"book"`/`"XXX"` for every non-Latin title, so ids stay distinguishable even before real transliteration exists.
- **Magic 26-letter/3-letter code-length ceiling with no graceful degradation** — [bookCode.ts:71](src/bookCode.ts#L71) (`GUIDE_LETTER_COUNT = 26`) and [chapterCode.ts:1-3](src/chapterCode.ts#L1-L3) (`CODE_LETTER_COUNT = 26`, `CODE_LENGTH = 3`) hardcode a 26-letter alphabet with a hard `throw` on overflow rather than any fallback (e.g. two-letter guide codes). Fine today, but a future "merge series" or "bulk import" feature would hit a hard ceiling with no caller-facing recovery guidance — `bookCode.ts` documents the cap reasonably well, `chapterCode.ts`'s overflow just throws.
- **Reliance on Obsidian's undocumented internal DOM/CSS** — [main.ts:244-286](src/main.ts#L244-L286) hardcodes selectors like `.workspace-drawer-vault-actions`, `.sidebar-toggle-button.mod-left/.mod-right`, `.inline-title`, `.view-header`, `.workspace-ribbon`, plus CSS variables `--frame-left-space`/`--ribbon-width`. [ToolsPanel.ts:41,51](src/view/ToolsPanel.ts#L41) also does `document.querySelector(".workspace-ribbon")` and reparents Obsidian's native ribbon node into the plugin's own view using `.side-dock-ribbon-action`. None of these are part of Obsidian's public API/theming contract — a future Obsidian release could rename or restructure any of them, silently breaking the "hide UI elements" settings or the Tools-panel ribbon relocation with no compile-time signal. This is inherent to the feature (hiding native chrome isn't otherwise possible) — worth centralizing the selector strings into one constants module so a breaking Obsidian update requires touching one place, and adding a smoke-test step against new Obsidian versions to the release process. `mountRibbon()` already guards with `if (!ribbon) return;`, which is good; the CSS-only hides in `main.ts` have no equivalent fallback.
- **No pop-out window support** — verified via grep: there is no `getActiveWindow`/`onWindowOpen`/`ownerDocument` handling anywhere in `src/`. Every style tag main.ts injects goes to the global `document.head`/`document.body`. Obsidian supports detaching a leaf into its own OS window (`WorkspaceWindow`) with its own `document` — none of this plugin's injected CSS (visibility rules, highlight colors, codex folder styling, ribbon relocation) will apply inside a pop-out window. Options: (a) accept and document as a known limitation (a real undertaking to fix — would need to listen for `workspace.on("window-open")` and inject a cloned `<style>` per window), or (b) at minimum add a comment so a future contributor doesn't assume it "just works."
- **Manually-kept-in-sync "mirror" schema (book.md mirrors series.md fields)** — `book-id-reference`/`book-title-reference`/`series-order-reference` are hand-synced across `book.ts` (`writeBookReferenceFields`, `syncBookReferenceFields`, `syncAllBookReferenceFields`) and referenced again in `migration.ts`. Every future "authoritative → mirror" field (e.g. a mirrored genre or word-count goal) requires re-deriving this same write-fan-out pattern by hand — there's no generic "denormalize field X into every book.md" helper. Acceptable at the current scope (3 fields); worth a small "mirror field" abstraction if more get added, given the copy-paste risk already visible in the 5x duplicated defaults-resolution pattern (§4).
- **Settings tab will not scale past ~2 stylable panel types without restructuring** — at 662 lines and ~30 individual settings, `StoryForgeSettingsTab.ts` isn't yet strained by logic complexity but by volume: every new panel type means another near-copy of the 4-setting header-style block (§4) and another manually-managed exclusivity pair (§4). Extracting a data-driven `renderStylableRowGroup(config)` would pay off before a third stylable panel type is added.
- **No pluggable "panel" abstraction** — `TopPanelOptions`/`BottomPanelOptions` ([TopPanel.ts:19-30](src/view/TopPanel.ts#L19-L30), [BottomPanel.ts:18-29](src/view/BottomPanel.ts#L18-L29)) each bundle ~8-10 callbacks owned by `StoryForgeView`. Adding a new panel type means growing `StoryForgeView.render()` and its instance fields directly — workable today, but every new panel is a bespoke addition to one view class rather than a pluggable unit.
- **Hardcoded UI strings with no central copy layer** — all setting names/descriptions/modal titles/row labels are inline literals scattered across 13+ files. Fine for a single-language plugin today; any future localization effort touches every file in this audit.
- **Fixed 4-level thickness enum** — `CodexFolderIndicatorThickness`/`CODEX_FOLDER_INDICATOR_WIDTH_PX` ([main.ts:19-26](src/main.ts#L19-L26)) hardwires exactly 4 named levels with fixed pixel values. Adding a 5th option or a free-form number later means touching the type, the settings UI, and this width map in three places.

## 7. Obsidian.md best-practice violations

- **Sentence case violations throughout the settings tab** — Obsidian's plugin style guide requires sentence case (only the first word/proper nouns capitalized) for setting names and headings. Verified directly: [StoryForgeSettingsTab.ts:91](src/view/StoryForgeSettingsTab.ts#L91) (`"Use Tools Panel"`), [:114](src/view/StoryForgeSettingsTab.ts#L114) (`"Colour Palette"`), [:127](src/view/StoryForgeSettingsTab.ts#L127) (`"Palette Mode"`) — and roughly 30 more `.setName(...)` calls plus several section headings ("Text Style", "Editor Text", "storyForge Interface Formatting", "Hide Obsidian Interface Elements") follow the same Title-Case pattern. One correct example already exists at line 81 (`"Reopen storyForge panel"`), showing the convention is known but not applied consistently. Fix: lowercase all but the first word/proper nouns throughout.
- **Redundant plugin-name heading in the settings tab** — verified: [StoryForgeSettingsTab.ts:76](src/view/StoryForgeSettingsTab.ts#L76) does `containerEl.createEl("h2", { text: "storyForge" })`. Obsidian guidelines explicitly discourage repeating the plugin's own name as a settings heading since it's redundant with the Community Plugins list entry. Should be removed.
- **Non-keyboard-accessible drag-and-drop** — [dragReorder.ts](src/view/dragReorder.ts) and [dragReorderTree.ts](src/view/dragReorderTree.ts) implement reordering purely via pointer events; there's no keyboard equivalent (e.g. focus row + arrow keys) anywhere, and drag handles have no `aria-label`/`role`/`tabindex` ([TopPanel.ts:87-89](src/view/TopPanel.ts#L87-L89), [BottomPanel.ts:119-120,164-165](src/view/BottomPanel.ts#L119-L120), [SeriesModal.ts:72-73](src/view/SeriesModal.ts#L72-L73)). A keyboard-only or switch-device user cannot reorder books/chapters/codex items at all.
- **Interactive controls built from non-focusable `<span>`s** — across [TopPanel.ts](src/view/TopPanel.ts), [BottomPanel.ts](src/view/BottomPanel.ts), [StatsPanel.ts](src/view/StatsPanel.ts), and the archive modals: `createSpan()` elements with a `click` listener carry `aria-label` (good) but no `tabindex`, `role="button"`, or `keydown` handling for Enter/Space — a screen-reader/keyboard user can perceive the label but can't activate the control. Fix: use Obsidian's `ButtonComponent`/`ExtraButtonComponent`, or add `tabindex="0"` + `role="button"` + keydown handling.
- **Custom SVGs duplicating stock Lucide icons** — [icons.ts:18-44](src/icons.ts#L18-L44) hand-rolls SVGs for concepts (inbox/archive, folder-plus, plus-square, filter, book, book-plus, calendar) that Obsidian's bundled Lucide set already covers by name (`setIcon(el, "box")`, `"folder-plus"`, etc.). Bespoke copies won't pick up Lucide's own visual-consistency updates when Obsidian upgrades its icon set, and add ~50 lines of maintenance burden. Icons with no obvious stock equivalent (the Codex "earth" icon, the dashboard variants) are reasonable to keep custom.
- **Non-`Setting`-API rows in modals** — [SeriesModal.ts:29-60](src/view/SeriesModal.ts#L29-L60), [ArchiveModal.ts:79-102](src/view/ArchiveModal.ts#L79-L102), [CodexArchiveModal.ts:61-79](src/view/CodexArchiveModal.ts#L61-L79) hand-roll rename-in-place inputs and list rows instead of `Setting`. Likely a defensible deviation since `Setting` doesn't fit an inline-rename row well — flagged for awareness, not necessarily a required fix.
- **No lint tooling** — no `lint` script in `package.json`, and no ESLint/Prettier config anywhere in the repo root. Obsidian's own plugin-review guidelines recommend `eslint-plugin-obsidianmd` specifically to catch this class of best-practice violation automatically. `tsc --noEmit` runs via the `build` script but there's no dedicated lint pass, and no CI workflow (`.github/workflows` doesn't exist) to enforce `test`/`build` on push/PR.

**What's already correct** (confirmed, not a criticism, but worth stating so nothing here reads as an unqualified negative): every `vault.on`/`workspace.on`/`metadataCache.on` call across the plugin is properly wrapped in `registerEvent` — no raw unregistered listener leaks were found. There's no direct Node `fs`/`path` or `require("electron")` usage anywhere in `src/`, matching `isDesktopOnly: false` in the manifest (mobile-compatible, and honestly declared). `onunload()` ([main.ts:204-217](src/main.ts#L204-L217)) correctly tears down injected style elements and body classes. `manifest.json`'s `id`/`name`/`minAppVersion`/version all follow convention and are internally consistent with `package.json` and `versions.json`.

## 8. Test coverage

Of 35 `src/*.ts` files (excluding `__tests__`), only 11 have a corresponding test file: `bookCode`, `chapterCode`, `codexPartition`, `codexTree`, `fingerprint`, `historyMath`, `ordering`, `orphanMatching`, `slug`, `titleNumbering`, `wordCount`. The other 24 — including every file under `src/view/` — are untested. The coverage that exists skews toward small, pure, easily-testable algorithmic helpers, which is reasonable prioritization. Two gaps stand out as disproportionate given what they do:

- **`writeGuard.ts` has zero tests** — verified directly (no `writeGuard.test.ts` in `src/__tests__/`). Its own code comments describe it as the one narrow module every plugin write funnels through, providing the non-destructive-write guarantee (`assertBackstagePath`, `ForbiddenWriteError`). This is arguably the single most safety-critical file in the codebase — its entire job is preventing writes outside the user's library/codex folders — and it has no automated regression protection.
- **`book.ts` (451 lines, the largest and most complex domain-logic file) has zero tests**, despite performing the actual frontmatter mutations that `writeGuard.ts` is meant to protect.

Given the plugin's core promise is non-destructive editing of an existing vault, these two are the highest-value gaps to close before the others.
